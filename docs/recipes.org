#+title: acmeid recipes -- spine tables for OTTR
#+author: johanw
#+date: 2026

* Why this exists

The historical workflow for our ontology source databases was:

1. =sqlite3= computes the rows.
2. The script exports CSV.
3. A separate Python/Emacs step mints identifiers, writes them back as
   another CSV.
4. OTTR templates read the CSVs.

That round-trip exists only because SQLite didn't know how to mint our
identifiers.  Once the =acmeid= extension is loaded, the spine table
is the registry, =acme_mint_id= is the registrar, and OTTR templates
read directly from the canonical SQL.  No CSV ferry, no risk of
re-minting on re-run.

The pattern in one sentence: /each spine table has a CHECK constraint
that the id verifies, and is populated by an idempotent insert that
mints only for rows that aren't already there./

* Loading the extension

** MSYS2 / Windows (canonical for this project)

#+begin_example
# Interactive sqlite3 in /mingw64/bin:
sqlite> .load c:/opt/acmeid/acmeid

# org-babel sqlite block:
:prologue ".load c:/opt/acmeid/acmeid"
#+end_example

Three small Windows notes:

1. Forward slashes are fine -- SQLite normalises them.
2. /No =.dll= suffix needed/ -- SQLite appends it automatically.
3. =-static-libgcc= is baked into the DLL, so =libgcc_s_seh-1.dll= is
   not required at load time.  =bcrypt.dll= is a system DLL.

** Linux / macOS

#+begin_example
sqlite> .load /usr/local/lib/acmeid
#+end_example

** When =.load= refuses

Some sqlite builds (notably the one shipped inside CPython on many
systems) are compiled with =SQLITE_OMIT_LOAD_EXTENSION=.  Use a
build that supports loading -- on MSYS2, =/mingw64/bin/sqlite3=
works out of the box; for Python, =pysqlite3-binary= does.

* The spine-table pattern

#+begin_src sql
CREATE TABLE dim_pitch (
    id    TEXT PRIMARY KEY CHECK (acme_verify_id(id)),
    label TEXT NOT NULL UNIQUE
);

-- Idempotent insert: mint only for rows that aren't already there.
INSERT INTO dim_pitch(id, label)
SELECT acme_mint_id('C','ex:',v.label), v.label
FROM (VALUES
    ('Pitch 1.0 mm'),
    ('Pitch 1.25 mm'),
    ('Pitch 1.5 mm')
) AS v(label)
WHERE NOT EXISTS (SELECT 1 FROM dim_pitch p WHERE p.label = v.label);
#+end_src

Running the same =INSERT= a second time adds zero rows; the spine is
authoritative and the ids are stable across runs.

* Worked example: =ISO_261_M_diam_pitch_class=

This reproduces the fastener-queries recipe without the CSV
round-trip.  Three spines plus one composite, all idempotent.

#+begin_src sql
-- 1.  Diameter spine
CREATE TABLE IF NOT EXISTS dim_diameter (
    id    TEXT PRIMARY KEY CHECK (acme_verify_id(id)),
    value REAL NOT NULL UNIQUE,
    label TEXT NOT NULL
);

INSERT INTO dim_diameter(id, value, label)
SELECT acme_mint_id('C','ex:', v.label), v.value, v.label
FROM (VALUES
    (3.0, 'M3'), (4.0, 'M4'), (5.0, 'M5'),
    (6.0, 'M6'), (8.0, 'M8'), (10.0, 'M10')
) AS v(value, label)
WHERE NOT EXISTS (SELECT 1 FROM dim_diameter d WHERE d.value = v.value);

-- 2.  Pitch spine
CREATE TABLE IF NOT EXISTS dim_pitch (
    id    TEXT PRIMARY KEY CHECK (acme_verify_id(id)),
    value REAL NOT NULL UNIQUE,
    label TEXT NOT NULL
);

INSERT INTO dim_pitch(id, value, label)
SELECT acme_mint_id('C','ex:', v.label), v.value, v.label
FROM (VALUES
    (0.5, 'P0.5'), (0.7, 'P0.7'), (0.8, 'P0.8'),
    (1.0, 'P1.0'), (1.25, 'P1.25'), (1.5, 'P1.5')
) AS v(value, label)
WHERE NOT EXISTS (SELECT 1 FROM dim_pitch p WHERE p.value = v.value);

-- 3.  Composite: ISO 261 M diameter-pitch class.
--     Composite's own id is minted in the same statement that joins
--     the spines for parent FKs.
CREATE TABLE IF NOT EXISTS dim_diam_pitch (
    id          TEXT PRIMARY KEY CHECK (acme_verify_id(id)),
    diameter_id TEXT NOT NULL REFERENCES dim_diameter(id),
    pitch_id    TEXT NOT NULL REFERENCES dim_pitch(id),
    label       TEXT NOT NULL UNIQUE
);

INSERT INTO dim_diam_pitch(id, diameter_id, pitch_id, label)
SELECT acme_mint_id('C','ex:', d.label || '×' || p.label),
       d.id, p.id,
       d.label || '×' || p.label
FROM (VALUES
    ('M3', 'P0.5'),  ('M4', 'P0.7'),  ('M5', 'P0.8'),
    ('M6', 'P1.0'),  ('M8', 'P1.25'), ('M10','P1.5')
) AS v(d_label, p_label)
JOIN dim_diameter d ON d.label = v.d_label
JOIN dim_pitch    p ON p.label = v.p_label
WHERE NOT EXISTS (
    SELECT 1 FROM dim_diam_pitch x
    WHERE x.diameter_id = d.id AND x.pitch_id = p.id
);

-- 4.  Verify idempotence: re-running the three INSERTs above adds 0 rows.
SELECT 'diameters: ' || COUNT(*) FROM dim_diameter;
SELECT 'pitches:   ' || COUNT(*) FROM dim_pitch;
SELECT 'composite: ' || COUNT(*) FROM dim_diam_pitch;
#+end_src

* Migrating an existing =fastener.db=

Tables created before the extension landed have no =CHECK= on the id
column.  Add one without dropping the table (and invalidating any
views that depend on it) by using =ALTER TABLE ... ADD COLUMN ...=
where supported, or this safer two-step:

#+begin_src sql
-- Verify every row first.
SELECT id FROM dim_pitch_legacy WHERE acme_verify_id(id) = 0;
-- (Should return zero rows.  If not, repair before continuing.)

CREATE TABLE dim_pitch_new (
    id    TEXT PRIMARY KEY CHECK (acme_verify_id(id)),
    label TEXT NOT NULL UNIQUE
);
INSERT INTO dim_pitch_new SELECT id, label FROM dim_pitch_legacy;
DROP TABLE dim_pitch_legacy;
ALTER TABLE dim_pitch_new RENAME TO dim_pitch;
#+end_src

⚠ =DROP TABLE= will invalidate any views that referenced the old
table; recreate them after the rename.  When the goal is just to
reseed values (not to change the schema), use =DELETE FROM dim_pitch=
inside a transaction instead.

* Handoff to OTTR

OTTR templates consume CSV exported from the spine:

#+begin_example
.mode csv
.headers on
.output dim_pitch.csv
SELECT id, value, label FROM dim_pitch ORDER BY value;
.output stdout
#+end_example

Two invariants make this safe:

1. The spine is the only place ids are minted.  Templates never call
   =acme_mint_id=; they only join on existing ids.
2. Because the spine is idempotent, re-running the SQL pipeline
   produces /the same/ CSV, byte-for-byte.  OTTR materialisations
   are therefore stable.

See =ottr-module-thread.org= for how the fastener templates consume
these CSVs.
